```sql
SELECT region, 
       TO_CHAR(sale_date, 'YYYY-MM') AS month, 
       SUM(sales_amount) AS total_sales,
       LAG(SUM(sales_amount)) OVER (PARTITION BY region ORDER BY TO_CHAR(sale_date, 'YYYY-MM')) AS last_year_sales,
       ROUND(((SUM(sales_amount) - LAG(SUM(sales_amount)) OVER (PARTITION BY region ORDER BY TO_CHAR(sale_date, 'YYYY-MM'))) /
```

---

### **1. Conflict: Choosing the Right Database for a Microservice**

**Scenario:**

* In the **market intelligence project**, one team member wanted to use **MongoDB** for the Reporting Service because of its flexible schema.
* Another team member argued that **PostgreSQL** was better because it supports **complex joins and transactions**.

**Challenges:**

* MongoDB is **good for unstructured data** but lacks strong ACID guarantees.
* PostgreSQL handles **complex queries well**, but performance could degrade with large datasets.

**Resolution:**

* After analysis, we **split the workload**:

  * Used **PostgreSQL** for **transactional data** (ACID compliance, strong consistency).
  * Used **Elasticsearch** for **fast analytics and reporting**.
* Implemented an **ETL pipeline** to sync data between the two.
* This **hybrid approach** provided both **data integrity** and **fast querying** for analytics.

---

### **2. Conflict: REST vs GraphQL for APIs**

**Scenario:**

* While designing the **Order Service**, some team members suggested using **GraphQL** because it **reduces over-fetching**.
* Others preferred **REST APIs** due to **simplicity and caching support**.

**Challenges:**

* **GraphQL** allows clients to fetch only needed data, but adds **query complexity**.
* **REST** is simpler but could lead to **multiple API calls** for related data.

**Resolution:**

* Used **REST** for core services (e.g., placing an order, fetching order status).
* Used **GraphQL** only for complex UI queries (e.g., fetching user, orders, and payments in a single request).
* This **balanced approach** kept APIs simple while improving client-side efficiency.

---

### **3. Conflict: API Gateway vs Direct Service Calls**

**Scenario:**

* One group suggested that **frontend should call microservices directly** to **reduce latency**.
* Another group argued that **an API Gateway** should be used to centralize authentication and request routing.

**Challenges:**

* Direct calls can reduce **latency**, but increase **frontend complexity**.
* API Gateway adds **an extra hop**, but provides **security, logging, and failover handling**.

**Resolution:**

* Implemented **Spring Cloud Gateway** for all external requests to enforce **security policies and rate limiting**.
* Allowed **internal microservices to communicate directly** where needed (e.g., Inventory & Order Service).
* This ensured **security without unnecessary overhead**.

---

### **4. Conflict: Managing Secrets – Environment Variables vs Vault**

**Scenario:**

* Some developers stored **database credentials** and **API keys** in **environment variables** for easy access.
* Others pushed for **HashiCorp Vault** to securely manage secrets.

**Challenges:**

* **Environment variables** are easy to set up but can be **leaked in logs**.
* **Vault** is secure but requires **additional infrastructure**.

**Resolution:**

* Used **Vault** for critical secrets (database credentials, OAuth tokens).
* Used **environment variables** only for **non-sensitive** configurations (app mode, logging levels).
* Integrated **Spring Cloud Config + Vault** so services fetch secrets securely.

---

### **5. Conflict: Handling Failures – Circuit Breaker vs Retries**

**Scenario:**

* The **Payment Service** sometimes failed due to third-party API downtime.
* Some developers wanted **automatic retries**, while others preferred a **circuit breaker pattern**.

**Challenges:**

* **Retries** can cause **thundering herd issues** if the service is truly down.

### **How do you optimize SQL queries for better performance?**

1. **Use Indexes:**

   * Create indexes on columns that are frequently used in `WHERE`, `JOIN`, `GROUP BY`, or `ORDER BY` clauses.
   * Avoid over-indexing as it may slow down `INSERT`, `UPDATE`, and `DELETE` operations.

2. **Avoid SELECT *:**

   * Always specify only the required columns in the `SELECT` statement to reduce data load.

3. **Use Joins Instead of Subqueries:**

   * Replace correlated subqueries with joins wherever possible for better performance.

4. **Optimize Joins:**

   * Use proper join types (`INNER`, `LEFT`, `RIGHT`, etc.) based on the requirement.
   * Ensure the joined columns are indexed.

5. **Limit Result Sets:**

   * Use `LIMIT` or `TOP` to retrieve only the necessary rows.

6. **Analyze and Refactor Queries:**

   * Use tools like `EXPLAIN` or `EXPLAIN PLAN` to analyze query execution plans and identify bottlenecks.

7. **Partitioning and Sharding:**

   * Partition large tables to distribute data effectively.
   * Use database sharding for horizontal scaling.

---

### **Explain normalization and denormalization with examples.**

1. **Normalization:**

   * Process of organizing data to eliminate redundancy and ensure data integrity.
   * Example: A customer table with repeated address data can be split into two tables:

     * `Customers (CustomerID, Name, AddressID)`
     * `Addresses (AddressID, Street, City, State)`
   * **Advantages:** Reduces redundancy, improves data consistency.
   * **Disadvantages:** Can increase complexity and require more joins.

2. **Denormalization:**

   * Combining tables to reduce joins and improve read performance.
   * Example: Merging `Customers` and `Addresses` tables into one:

     * `Customers (CustomerID, Name, Street, City, State)`
   * **Advantages:** Faster read queries.
   * **Disadvantages:** Increases redundancy, risk of data inconsistency.

### **How do you handle transactions in Java using Spring JDBC or JPA?**

* **Spring JDBC:**

  * Use `@Transactional` annotation to demarcate transactional boundaries.
  * Example:

    ```java
    @Transactional
    public void performTransaction() {
        jdbcTemplate.update("INSERT INTO accounts (id, balance) VALUES (?, ?)", 1, 1000);
        jdbcTemplate.update("UPDATE accounts SET balance = balance - 100 WHERE id = ?", 1);
    }
    ```

* **Spring JPA:**

  * Transactions are managed automatically by Spring's `EntityManager`.
  * Example:

    ```java
    @Transactional
    public void performTransaction() {
        Account account = accountRepository.findById(1);
        account.setBalance(account.getBalance() - 100);
        accountRepository.save(account);
    }
    ```
  * Ensure the transaction manager is correctly configured in your application.

### **Write an SQL query to find the second-highest salary in a table.**

Assume the table is `Employees` with columns `EmployeeID` and `Salary`.

1. **Using LIMIT (MySQL):**

   ```sql
   SELECT DISTINCT Salary 
   FROM Employees 
   ORDER BY Salary DESC 
   LIMIT 1 OFFSET 1;
   ```

2. **Using Subquery:**

   ```sql
   SELECT MAX(Salary) 
   FROM Employees 
   WHERE Salary < (SELECT MAX(Salary) FROM Employees);
   ```

3. **Using `DENSE_RANK` (SQL Server, Oracle, PostgreSQL):**

   ```sql
   SELECT Salary 
   FROM (
       SELECT Salary, DENSE_RANK() OVER (ORDER BY Salary DESC) AS rank 
       FROM Employees
   ) ranked 
   WHERE rank = 2;
   ```
